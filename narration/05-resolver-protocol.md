Chapter five. The resolver protocol.

We keep saying "the resolver" as if it were a known thing. So let's make it precise. What, exactly, is a resolver? The answer is delightfully plain: a resolver is whatever satisfies the contract. There's no base class to inherit from, no ceremony to perform. Implement the right members with the right shapes, and you are a resolver. That's it.

And the contract is small. Twelve members in total — four properties and eight methods. That is the entire surface a resolver has to present to the platform. Let me give you the properties first, because they're simple. A name, unique, like "dot-graph." A version. A human-readable description, the one that shows up when you list the resolvers. And a single yes-or-no flag — does this resolver support resume? Can a paused run pick back up, or not?

Then the eight methods. And the nicest way to understand them is to hear them as answers to questions the platform asks, in order, over the life of a run. Before the instance even exists, the platform asks: what do I need to collect from the user to start this? Then, while it's building the container: which repositories do I clone, and how do I set up git? And: what extra setup does the container need? Then comes the big one — just "do the work." After that, the platform may interrupt with: the user answered your question; or, the user sent you a message to steer; and finally, when someone asks for a graceful stop: wind down, cleanly. Each method is the resolver's answer to one of those questions, asked at the right moment.

Of all of them, one method matters most, and it's the one that simply says "do the work." It's called run, and it receives three things — and that triad is the resolver's entire world. First, the config: everything about this particular instance, the inputs, the working directory. Second, the session factory: the engine for spinning up Amplifier sessions cheaply — this is how the resolver actually drives a model. And third, the emitter: the single channel for events, for state snapshots, for asking the human a question. Config, session factory, emitter. Give a resolver those three, and it can do anything.

Now, why structural instead of nominal? Why no base class? Because a structural contract means a resolver can be written in any style, by anyone, as long as it presents the right shape. Small, stable contracts age better than base classes, which tend to accumulate baggage over the years. Anything with these twelve members simply is a resolver. There's nothing to subclass and no inheritance chain to fight.

A word of honesty here, because this masterclass values it. There's a spot where the written design documents and the running code disagree. The old documents describe run as returning a result object that names the outcome. The code has moved past that. Resolvers now run as separate worker processes, and they signal their ending through the event vocabulary, not through a return value. The idea is unchanged — a run still reaches completed, or paused, or failed. But the mechanism of reporting it is now an event. And the rule, whenever a document and the code disagree: the code wins.

Step back, and the protocol is really just the bright line drawn as an interface. The platform owns the lifecycle, the registry, the structural validation, the container, the routing of responses. The resolver owns all the meaning — what to ask, what it means, which repositories, when to block, what counts as done. Same split as always, now expressed as a set of method signatures.

And one last elegant touch: registration is validation. You install a resolver as its own package and add it, and that act of adding is the check — if it doesn't satisfy the contract, it doesn't register. The catalog is the single source of truth. If it's in the catalog, it validated. No half-built resolver ever surfaces by accident. With the contract clear, the next question is where a resolver actually lives when it runs — and that's the sealed container.
