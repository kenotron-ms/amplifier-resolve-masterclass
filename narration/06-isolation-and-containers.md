Chapter six. Isolation and the container model.

So we know what a resolver is. Now, where does it run? And the answer matters more than it might seem, because of what a resolver actually is: code the platform did not write, driving a model that does unpredictable things, often against real repositories with real credentials. The only safe place to run something like that is a sealed box. So the container is the isolation boundary — and the filesystem inside it is the coordination bus. This chapter is about both of those at once.

Start with the box itself. Worker containers run on one of two engines: Incus, the primary, or Docker, the fallback — and the platform detects which is present and chooses at runtime. You, writing a resolver, never see the difference. The orchestrator asks for "a container" and gets one. That's deliberate: coding to an abstraction means a brand-new runtime someday is a new implementation, not a platform rewrite. It's the same structural-contract instinct you saw with resolvers, applied one level down.

Even the container names are doing work. They're not cosmetic — they're a coordination protocol. The main container is named for the instance. Its git sidecar is named for the instance, with "gitea" on the end. An ephemeral sub-container gets the instance name plus a unique suffix. And because the name encodes the relationship, the watchdog and the cleanup sweep can find every container belonging to an instance without keeping a separate database anywhere. The name is the record.

Now the part I want you to really hold onto — the filesystem as the coordination bus. Inside every worker container there's a directory, the dot-resolve directory, and it works like a shared mailbox with two writers and very strict rules about who writes what. The host writes the configuration in. It writes responses to questions in. It writes steering messages in. The resolver, through its emitter, writes the other direction — the event log, the state snapshot, the status checkpoint, the schemas for any questions it's asking. And then each side reads what the other one wrote. The host reads the resolver's events; the resolver reads the host's responses. Two writers, a clean ownership split, no collisions.

Why a filesystem, and not a socket or a message queue? Because a file-based bus is inspectable — you can literally tail the event log and watch the run think. It's append-only where that matters, atomic where it must be, and — this is the quiet superpower — it survives a daemon restart with no reconnection logic at all. The platform observes a run simply by reading files a process happens to be writing. The only cost is discipline: one writer per file. And the ownership split is what enforces it.

Then there's safety, and here the masterclass earns its seriousness, because this design was paid for in a real incident — more than four thousand runaway test processes, over a hundred gigabytes of memory, the system forced to start killing things. Out of that came three independent layers of defense, and no single layer is trusted to be enough. Layer one, the kernel limits, set when the container is created: a hard cap on the number of processes, a hard memory ceiling with no swap, a CPU quota. That's the last line — even if everything above fails, the kernel contains the blast. Layer two, a process guardian running inside the container as a hook. It watches process counts and kills runaway test processes before each tool call. And because it's a hook, not a tool, the model cannot see it, forget it, or decline to use it. Layer three, a periodic watchdog running outside, watching process counts and memory and the total lifetime of the instance, with a hard twelve-hour maximum.

Three layers, each catching a different failure. The guardian stops most fork bombs before they ever reach the kernel limit. The watchdog catches the slow-burn exhaustion that happens between tool calls, when the guardian isn't looking. And the kernel limit catches anything that slips past both. The redundancy isn't waste — it is the design. Containment is the one property the platform may never compromise, so it's defended in depth.

The box is sealed and the bus is laid down. Next, the messages that flow across it — the event vocabulary that turns a run into something you can read.
