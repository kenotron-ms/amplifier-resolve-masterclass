Chapter seven. The event vocabulary.

If you remember only one technical chapter from this whole masterclass, make it this one. This is the observability contract — the small, fixed set of events every resolver agrees to emit, so that a run becomes legible to the platform without the platform ever understanding what the run is actually doing inside. And there's one invariant in here that is the difference between a platform that works and a platform that silently hangs. We'll build up to it.

First, a distinction that shapes everything: this is a vocabulary, not a schema. Hear the difference. A schema would lock down the exact shape of every event's payload, so that adding a single field becomes a breaking change. A vocabulary is looser and wiser. It fixes only the names — the set of event names that a cross-resolver consumer is allowed to rely on, and a handful of fields inside each one. Underneath, resolver authors stay free to add whatever keys they want, and anyone who doesn't recognize those extra keys simply ignores them. So the platform pins down the names it interprets, and lets the payloads grow freely. It's forward-compatible by construction.

There are eight events, and they all live under one prefix — "resolver," followed by a colon. You don't need to memorize the list, but you should feel its arc. A run announces itself when it's created. It marks each phase boundary as it moves. It can report progress, when there's something worth reporting. It announces each artifact it produces — a branch, a commit, a pull request, a file. Two of the events fire automatically when the resolver asks a human a question — one when the question goes out, one when the answer comes back — so the author gets those for free. There's an optional health signal for when a run is under stress. And then, last, the one that matters above all the others: completed.

Now let me draw that invariant in the air, because this is the keystone. Picture every place a run can end. The clean success at the bottom of the happy path. The failure when something goes wrong. And — this is the one people forget — the exception, the path nobody planned for, the surprise error halfway through. Here is the rule. At every single one of those exits — success, failure, and exception alike — the resolver must emit the completed event. Every exit. No exceptions to "every exit." Because the platform's notification loop is waiting on exactly that event to know the run is over. If even one exit path forgets to emit it, the loop just waits — until a fallback eventually fires. And one missed exit path is one silent hang. So when you read the contract — completed event first, then process exit as a fallback, then a watchdog timeout as the last resort — understand what the first line is buying you: a clean, structured ending whenever the resolver cooperates. Your whole job, as a resolver author, is to make sure it always cooperates. Emit it at every return. Emit it in every except. Or the platform, rightly, assumes the worst.

And a word on history, because it sharpens the point. The old way of signaling completion was to print a magic string into the output and have the platform watch for it. That was removed, on purpose. A typed event can't be printed by accident from some unrelated log line. It carries structured outcome and error as real data, not text to be grepped. Do not bring the magic string back in that role.

There's one more trap worth naming before we go, and it's about state. Alongside the events, a resolver keeps a live snapshot of its state that the viewport renders. And there are two ways to write that snapshot, easy to confuse and painful when you do. One of them replaces the whole document — everything not in this call is simply gone. The other one merges — it reads what's there, overlays the new keys, and writes it back, preserving everything accumulated so far. The rule to carry: partial updates merge; complete snapshots replace. And the platform does the merging itself, on the resolver's behalf, precisely so that a resolver sending small incremental updates — a line of narrative here, a roster change there — doesn't accidentally wipe out everything it sent before. Merge for the trickle; replace for the full picture.
