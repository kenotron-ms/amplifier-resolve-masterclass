Chapter nine. The session engine.

A resolver decides strategy. But to carry that strategy out, it has to run Amplifier sessions — and running sessions well, cheaply, and observably is a hard problem. So the platform solves it once, in a shared toolkit, and every resolver inherits the answer. This is the layer that sits underneath all three of them.

Start with a simple economy. Preparing a bundle — downloading modules, resolving dependencies, composing configuration — is slow. Creating a session from a bundle you've already prepared is fast. So the session factory separates the two deliberately. You prepare once, at startup, and pay that cost a single time. Then you create many sessions, cheaply, one per phase. A resolver that runs five phases would otherwise pay the full setup five times over. Prepare once, create many. That's the whole trick, and it's what makes multi-phase work practical.

Now, a single resolver run is rarely a single session. The dot-graph resolver runs a session per node. The orchestrator spawns workers. The understudy dispatches verifiers. If you want observability to mean anything, all those scattered sessions have to stitch back to the run that spawned them. That's the job of the parent id. When the factory mints a new session, it stamps the parent's identity onto it, so the whole tree links together. Pass the parent id, and you get a properly stitched family. Omit it, and you get a flat, disconnected list. Same code, two outcomes.

At the heart of the toolkit is the event emitter — one per instance, threaded through the entire run. Its discipline is worth saying out loud. When it emits an event, it appends a single line to the log, flushes it immediately so you can watch the run live, and — this matters — it never raises. A logging failure must never crash a run. When it writes state, it writes the whole document atomically, so a reader never catches a half-written file. And here's the quiet, beautiful part: if there's no working directory, every method becomes a silent no-op. The same resolver code runs identically in a real container, writing files, and in a unit test, writing nothing — with no branching, no special case. Observability that can never be the thing that breaks you.

Two higher primitives sit on top, and each encodes a hard-won lesson. The first is the disposable session: spin one up for a bounded sub-task, get an answer, and always tear it down in a finally block. Sessions are cheap and throwaway by design. The second is the antagonistic review, and this one is the quality keystone. It captures the git diff — just the changes, not the whole repository — hands that and the spec to a separate reviewer, and tells it to go find problems. The insight is that an agent reviewing its own work rationalizes; an agent shown the entire codebase drowns. But a fresh reviewer, bound to only this diff and told to be adversarial, produces findings you can act on. Quality comes from an adversarial reviewer bounded to the change, not from trusting the author.

And finally, the question of done. Judgment decides strategy; deterministic checks decide done. The toolkit runs build, then test, then lint — each one required to exit clean — plus a regression check so a change can't quietly add new failures. A model will happily declare success. A failing test suite will not. Routing the final go-or-no-go through real exit codes turns "I think it works" into "the tests pass." That's where autonomous work earns its trust: the finish line is executable.

So that's the engine. One shared solution to running sessions. Next, we'll watch three radically different strategies built on top of it.
