Chapter eleven. Viewports and the single-page app.

Three resolvers as different as a compiled pipeline, a delegation manager, and a governed apprentice each need a face — a little interface that shows what that particular resolver is doing. But remember the bright line: the platform must not understand any resolver's domain. So how does each one get a custom face without the platform learning what a pipeline or a worker or a journey even is? The viewport system is the answer. A resolver ships its own user interface as a plugin, and the web app loads it without ever knowing what it draws.

Start with the app itself. It's a shell, and nothing more. It provides a stable host — navigation, sign-in, the event stream, the state poll — and into that host, resolver-authored viewports are injected at runtime. The shell knows exactly one thing about any resolver: the shape of the props it hands across. It knows nothing about pipelines, or workers, or journeys.

Each resolver may ship a single bundled component, and the platform serves it on request. When you open an instance, the app derives the right address, quietly fetches the bundle first — so that a resolver with no custom face simply returns nothing, instead of throwing a red error — and then loads it. And it's careful to fully tear down one viewport before mounting another, so a dot-graph view never lingers and tries to read graph data from an orchestrator run. A broken or mismatched viewport becomes a contained failure, never a crashed app.

Now the heart of it: the props contract. A viewport is a pure function of its props. Data flows in — the events so far, the latest state, the status, any pending input requests. Capabilities flow out — send a message, fetch a large file on demand. And that's the entire surface between the shell and the viewport. The viewport never talks to the platform directly. It only receives what it's given and calls what it was handed. That narrow, fixed boundary is the seam that lets the shell and every viewport evolve independently. The shell can change how it fetches state, swap its transport, even sandbox a viewport — all without touching viewport code. And a viewport can be tested by simply passing it props.

There's one piece of deep machinery worth naming, because it looks like incidental complexity and is exactly the opposite. The shell and every viewport must share one single copy of React. If a viewport bundles its own, hooks crash instantly, because each copy keeps its own internal bookkeeping. So React is externalized — pinned to one shared instance, in production through an import map, and in development through a careful shim. Two copies of React is the single most common way a plugin user-interface architecture dies. Do not simplify that shim away.

And if a resolver ships no face at all? It still gets a usable one for free. A generic fallback reads only the universal contract — events, state, status — and renders a sensible default: a pulse of activity, a current-task readout, and cards that infer their own display type from each piece of state. A custom viewport is an upgrade, never a prerequisite. That keeps the barrier to writing a new resolver low.

One last inversion, and it's a subtle one. You'd expect the resolver to define its state and the viewport to display whatever it gets. It's the other way around. The viewport owns the schema; the resolver conforms. The viewport is the thing that actually renders the state, so it's best placed to declare what shape it needs — and the resolver writes to match. Consumer defines, producer conforms. That keeps a resolver's state and the one thing that displays it living together, so they can never quietly drift apart.

The shell stays constant. The viewports are the free variables. It's the bright line from the very first chapter, carried all the way out to the browser.
